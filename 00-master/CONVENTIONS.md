## 1. .md 모듈 헤더 양식 (필수)

모든 모듈 .md 최상단에 다음 YAML 프론트매터를 둔다.

**예외 (00-master/ self-exempt)**: `00-master/MASTER-PLAN.md`, `00-master/CONVENTIONS.md`, `00-master/CHANGELOG.md`, `00-master/PROGRESS-TRACKER.md` 4개 마스터 파일은 frontmatter 면제. MASTER-PLAN/CONVENTIONS는 frozen SSoT (status 개념 없음 — 변경은 CHANGELOG 기록 + PM 승인), CHANGELOG/PROGRESS-TRACKER는 living document (지속 갱신 — `status: done` 자체가 어색). 모든 hierarchy 1~6 모듈은 본 frontmatter 룰을 따른다.

```markdown
---
module: H1-cha-serin
hierarchy: 1
depends-on:
  - 00-master/MASTER-PLAN.md
  - 00-master/CONVENTIONS.md
  - 02-characters/goo-yunmo.md
outputs:
  - 차세린 캐릭터 시트
  - 차세린 스프라이트 표정 8종 명세
  - 차세린 루트 트루엔딩 트리거 조건
status: draft
---
```

`status` 값: `draft` → `in-progress` → `review` → `done`

## 2. 작업 단위 원칙

- **하나의 .md = 하나의 책임 (Single Responsibility)**
- 다른 모듈의 영역을 침범하지 않는다
- 작업 결과는 정해진 파일에만 쓴다
- 다른 모듈의 .md를 수정해야 한다고 판단되면, 작업 끝에 "변경 제안" 섹션에 기록만 하고 직접 수정 금지

- KEEP (실연출): "마지막 🥺🥺 후 2초 정지", "잠깐 멈춘다(실제 멈춤)"

#### 4. "본심" 남발 제한

### 3.7 톤 보상 매트릭스 (2026-04-30 신규, 사용자 직접 명령)

다섯 히로인 보상 함수가 동일 공식(KEY +10 / 묘사 +5 / 가벼운 답 0~-2)으로 통일돼 캐릭터 분화가 시스템 레벨에서 지워지는 문제를 해결한다. 한 히로인의 정답이 다른 히로인의 오답이 되는 자리를 시스템적으로 만든다.

#### 1. 톤 5종 (선택지 라벨에 1개 박는다)

- `mature_serious` — 어른·진중·학술 톤 (예: "선배님, 무리 마시고 천천히 하세요")
- `warm_supportive` — 정중·케어·존대 톤 (예: "괜찮으세요? 식사는 하셨어요?")
- `direct_friendly` — 동기·반말·솔직 톤 (예: "야 너 이번 시험 족보 알아?")
- `playful_casual` — 가벼움·반말·장난 톤 (예: "ㅋㅋㅋ 너 진짜 웃기다")
- `bright_forward` — 밝음·직진·이모지 톤 (예: "선배 오늘 진짜 멋있었어요!")

작가는 5종만 외우면 된다. 점수는 매트릭스(아래 §3)가 자동 계산.

#### 2. 신표기법

```
[CHOICE:
  1) 무리 마시고 천천히 하세요   <tone=mature_serious, key=true, descriptor=quiet_care>

  3) ㅋㅋㅋ 너 진짜 웃기다         <tone=playful_casual>

```

#### 4. KEY 라벨 [Δ 2026-04-30 Step 4 갱신 — H3 시간대 무관 + H4 대면 KEY]

선택지에 `key=true` 마커가 있으면 해당 히로인의 KEY 톤일 때 묘사 보너스 +5 추가 가산. KEY 톤 매핑:

- H2: `direct_friendly`
- H3: `warm_supportive` ※ **시간대 무관** (2026-04-30 Step 4, PM 결정). H3_NIGHT_DELTA는 일반 자리에만 보정 적용. KEY 자리는 낮/밤 모두 묘사 보너스 가산.
- H4: 다음 두 자리 KEY 인정 (2026-04-30 Step 4 + 2026-05-05 옵션 B):
  - 미니게임 자리: `mechanism:h4_reply_speed` + 통과(replyTimeMs < 15s) + isKey
  - 대면 KEY 자리: `mechanism:h4_facing_key` + isKey + `tone:warm_supportive` → 묘사 보너스 +5
    - **2026-05-05 옵션 B**: H3 KEY와 동일 톤(warm_supportive) 충돌을 방지하기 위해 명시 마커 도입. 마커 누락 시 H3 KEY로만 라우팅.
- H5: `bright_forward` (시간대 무관)

#### 5. 보조 메커니즘

- **H3 시간대 갭**: 같은 톤도 낮/밤에 다른 점수. `[SCENE: time=night]` 마커 시 `warm_supportive` -2 / `playful_casual` +4 / `mature_serious` -4. 자세한 룩업은 `06-engine/toneMatrix.ts` (Step 2).

#### 6. "정답이 오답되는" 자리 — 작가 가이드

같은 선택지 텍스트가 5명에게 동시에 다른 점수가 된다. 5지선다 자리에선 5명 톤을 골고루 분포시키는 것을 권장:
- 자리 1: `mature_serious` (H1 정답, H5 오답)
- 자리 2: `warm_supportive` (H3 정답)
- 자리 3: `direct_friendly` (H2 정답)
- 자리 4: `playful_casual` (H1 페널티 자리, H5에는 +5)
- 자리 5: `bright_forward` (H5 정답, H1에 페널티)

5명 모두에 균형 분포한 5지선다는 캐릭터 학습 기회를 주고, "한 톤만 골라서 다 풀기" 우회를 차단한다.

#### 7. 자체점검

각 시나리오 작가 메모에 다음 한 줄 추가:

- KEY 라벨이 정확한 히로인 KEY 톤에 박혀 있는지 (예: H1 KEY인데 `direct_friendly`로 박혀 있으면 오류)
- H4 미니게임 자리에 `mechanism=h4_reply_speed` 마커 누락 없는지
- H4 대면 KEY 자리에 `mechanism=h4_facing_key` 마커 누락 없는지 (마커 없으면 H3 KEY로 라우팅됨, 2026-05-05 옵션 B)
